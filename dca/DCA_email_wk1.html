Okay, I've kicked things off. I've set up a Chef Automate server in our datacenter, and have "bootstrapped" all of our servers. They can report into Chef Automate, but Chef is not yet making any changes.
To start, our Chef rep suggested we use the Audit Cookbook to scan our nodes, and see how securely they're set up. I haven't yet defined our security requirements, but I'm told the Automate server contains a "Baseline Security" scan we can use for the time being to get a baseline.
I've installed Chef's Development Kit on our shared workstation, and you'll be using two tools primarily:

knife: A commandline utility for querying our fleet, and applying changes to their configs.
kitchen: A testing utility we can use to evaluate changes before we run them on our servers.

Here's what you need to do:
  1. Log in to Chef Automate and install the "DevSec Linux Security Baseline" from the "Profile Store" in the "Compliance" tab
  2. You should find a copy of the dca_baseline cookbook in the dev workstation's home directory. Navigate to its folder, and run "kitchen converge". This will launch a test instance using our base image, and run that security scan. When complete, you should be able to see the results in the Automate UI.
  3. Update "dev1" and "dev2" to run audits. This is done by updating their "run list", and can be done like so with knife: "knife node run_list add NODE_NAME '''recipe[dca_baseline]''' "
  4. If the test works, ensure that the "dca_baseline" cookbook is part of our development machines' run lists, and run the chef client on them. You can do this for the whole dev environment with the command "knife ssh 'chef_environment:development' 'sudo chef-client'". This will search for all development servers, and run the chef client on them over SSH.
That'll be it for week 1! If we discover that any scans are failing, we'll strategize on correcting those issues in Week 2.
